## Architecture decisions

- Run-queue retry budget is a **shared contract** (`@attest/contracts/queue.ts`: `MAX_RUN_ATTEMPTS=3`, `RUN_BACKOFF_MS`), not a worker-local constant: BullMQ's retry/backoff is set by the **producer** (backend) at `queue.add`, while the worker decides "is this my last attempt?" using the same `attemptsMade+1 < attempts` gate BullMQ uses internally. Both reading one constant keeps the §7.5 policy coherent across the boundary.
- Worker lifecycle terminal states: a run resolves to `completed` (attestation persisted, any verdict incl. `inconclusive`) or `canceled` (allowlist denial, missing app, or a final-attempt worker fault). `run.lifecycle` has no `failed` state by design - a verdict `failed` is a *product result* on the attestation, not an operational error (`[tech-arch §4.3, §5.1]`); operational faults use `canceled` + `run.error`.
- Worker handler is engine-injected (`process-job.ts` takes `run: (input, job) => Promise<Attestation>`), so the §7.5 retry/lifecycle/allowlist logic unit-tests with fakes and never pulls puppeteer/openai/db drivers; `index.ts` is the only place that binds the real `runAttestation` + concrete adapters. Keeps the testable policy separate from the composition root.
- Auth/tenancy table ownership: **BetterAuth owns `org`/`user`/`member`/`session` etc.** (`organization` plugin), generated into `packages/db/src/schema/auth.ts` by `@better-auth/cli generate`; Attest's domain tables hold `org_id` as a real FK → `organization.id`. Why: don't hand-roll auth shapes BetterAuth dictates (no drift; re-run the CLI after adding any auth plugin). The BetterAuth instance lives in `apps/backend` (`[arch §3.3, §6.1]`), not `packages/db` (data layer stays runtime-free). MCP service keys are a **custom domain table** (`app_key`, org + app scoped), not BetterAuth's per-user apiKey plugin, since the scope model differs (`[arch §6.2]`).
- Resolved `[arch §13]` open questions with provisional defaults - see `[tech-arch §7]`: no plan cache (MVP), SSE streaming, one fresh browser context per job, allowlist enforced at enqueue + re-checked in worker, ≤2× retry then `inconclusive`, envelope encryption (KMS hosted / operator-key self-hosted).
- Why: favor correctness and the clean-room/isolation guarantees over premature optimization; each default is reversible behind an interface or flag.
- Billing (`ee/`): hybrid subscription + credits via Dodo Payments (Merchant of Record, handles global tax). Local credit ledger is authoritative for real-time enqueue gating; Dodo is payment rail + grants via webhooks (Dodo's native credits rejected: async ~1-min deduction unfit as a gate). Costs documented as directional ranges with explicit ~60-75% margin multiplier; raw `UsageEvent` (runs+minutes+steps+model-cost) is the tunable source of truth. BYOK runs cost infra-only, kill the §7.3 margin risk (`[tech-arch §13]`).
- Models: OpenRouter is the gateway (one OpenAI-compatible API, one key, all providers) behind `adapters/model`; user picks the model per agent role (planner/judge/resolution) in the dashboard, defaults follow the tiered strategy. BYOK = user's OpenRouter key; self-hosters can point at a local OpenAI-compatible endpoint. Per-run cost now tracks user model choices (`[tech-arch §3.3]`).
- Resolution adapter is a **deterministic Puppeteer-native ladder**, NOT Stagehand (deviation from `[tech-arch §3.2, Tech stack]` which names Stagehand). Why: Stagehand is Playwright-based and owns its own page/session, which clashes with the Puppeteer browser adapter that owns the live page; the two cannot share one context cleanly. The ladder (a11y→text→aria→role) covers the deterministic rungs `[§7.1]` and is dependency-light; the resolution interface is unchanged, so Stagehand + a cheap-model fallback can be swapped back in later as the heavier rung. Stagehand dep not installed. **Confirm/fix the §3.2 + Tech-stack Stagehand reference on the next adapter doc pass.**
- Concrete adapters live in `core/adapters/*` but are **deliberately absent from the adapters barrel** (`adapters/index.ts` exports interfaces only): putting them there would force puppeteer/openai/aws-sdk onto every `@attest/core` importer (e.g. backend, which needs only types). **Resolved**: each concrete adapter is exposed via a **per-adapter subpath export** in `core/package.json` (`@attest/core/adapters/browser/puppeteer`, `/model/openrouter`, `/resolution/dom-ladder`, `/storage/{disk,s3}`), targeting the emitted `dist/.../*.{js,d.ts}`. The composition root (worker) imports exactly the factories it needs; any other consumer pulls only the one SDK it touches (e.g. backend evidence resolution → `storage/s3` only, no puppeteer). Chosen over a single `@attest/core/worker` entrypoint because that would drag all engine SDKs onto any importer. Interface barrel untouched (subpaths point straight at the file).
